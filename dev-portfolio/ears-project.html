Audio Dichotics: Designing and Building a Web-Based Auditory Training Program

As part of a cross-functional team, I helped design and develop a web-based training tool aimed at improving auditory processing skills. Over the course of four sprints, we built an interactive site where users could complete audio-based training exercises and receive feedback. My role focused on front-end development and interface consistency.

Landing page of the auditory training program

The Problem

The original web tool needed to be expanded with additional auditory training content, but lacked a user-friendly UI and functional structure. Our task was to design and build new training pages, maintain visual consistency, and ensure everything worked smoothly across sessions.

The Process (Sprints Overview)

Sprint 1: Project Planning

We mapped out our approach and met with the original development team to understand their codebase. Our goal: expand the existing system without breaking its structure.

Key takeaway

Collaboration early on helped avoid friction later.

Sprint 2: Lo-Fi Wireframes + Feedback

Using Balsamiq, we created our first draft low-fidelity wireframes and presented them for feedback. These initial wireframes focused on page layout and test input forms.

Challenge

Without audio files, we had to focus on layout, interaction, and architecture first.

Updated Wireframes

After we were given feedback on our initial wireframes, we started our second draft. These wireframes focused on user-specific content, streamlined training flows, improved result visibility, and new administrative tools, such as a client edit page.

Sprint 3: Development Begins

With audio files finally available and wireframes approved, I built the Week 1 training pages. I focused on ensuring the pages were functional and visually consistent with the original site's style guide.

  • My contribution: Web page development + interaction functionality
  • Consistency goal: Reuse visual language from the original site to avoid user confusion.

Challenge

One of the algorithms for the RDDT tests was giving incorrect results. After some digging, I realized it was a calculation bug—not a logic issue. I corrected the math and got the test running correctly.

Completed training page interface

Sprint 4: Final Pages + Polish

With an extended sprint, we added all remaining training pages and core functionality like result tracking. Though we initially used temporary audio files, we later updated the site once we received the final versions.

  • New features added: Results landing page
  • New features added: Test score saving functionality

Key Challenges + Solutions

  • Missing assets early on: We adjusted timelines and prioritized wireframing.
  • Buggy scoring algorithm: Fixed a tricky calculation issue causing inaccurate results.
  • Team coordination: Balancing updates with the original development team's expectations meant a lot of alignment and double-checking.

Outcome & Learnings

The training platform is now live and being used to help users improve auditory processing through structured, weekly exercises. I learned how to build within an existing system, how to communicate design intent across dev teams, and how to debug under pressure when core features aren't working.
